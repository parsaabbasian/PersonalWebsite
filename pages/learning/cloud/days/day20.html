Day 20

API Auth & JWT

Moving beyond sessions to stateless, token-based security for scalable modern backends.

1 The Shift to Token-Based Auth

Unlike web apps that use cookies and sessions (stateful), APIs must be Stateless. The server shouldn't remember the client; instead, the client presents a "digital passport"—a Token—with every request.

Sessions (Stateful)

Stored in server memory or DB. Hard to scale across multiple servers.

JWT (Stateless)

Self-contained info. Verified instantly by any server instance using a secret key.

2 Anatomy of a JWT

A JSON Web Token consists of three parts separated by dots. It is Encoded, not encrypted, meaning anyone can read the data, but only the server can verify it wasn't tampered with.

header. # Algorithm & Token Type
payload. # User ID, Expiry, Roles
signature # The Cryptographic Proof

3 The Authentication Cycle

1

Login & Issue

Client sends credentials; Server returns a signed JWT.

2

Bearer Token Header

Client attaches Authorization: Bearer <token> to all requests.

3

Verify & Authorize

Server validates the signature and grants access based on payload data.

Refresh Token Strategy

Since JWTs are stateless, you cannot easily "revoke" them. We use short-lived Access Tokens (15 min) and long-lived Refresh Tokens to keep users logged in securely.

Security Alert: Never store sensitive data (passwords, social security numbers) in a JWT payload. It is Base64 encoded and easily readable by anyone.
