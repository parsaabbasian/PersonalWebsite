Day 8

Forms & User Input

Mastering the flow of data from browser to server while implementing industry-standard security and validation.

1 The Golden Rule: Never Trust the Browser

Every piece of user input is untrusted. Malicious users can bypass your HTML, modify your JavaScript, or use tools like curl to send fake requests.

Backend Mindset:

Always assume form data is wrong, missing, or tampered with until it is proven valid by the server.

2 The Form Submission Flow

1. User fills form & Submits

Browser sends a POST request with encoded data.

2. Flask receives request

Data is stored in request.form as raw strings.

3. Server-Side Validation (The Gatekeeper)

Flask-WTF checks for length, format, and security tokens.

4. Action & Response

Data is saved to DB, and the user is redirected or shown errors.

3 Client-Side vs Server-Side

Client-Side (HTML/JS)

User Experience Only

Immediate feedback for users. Prevents accidental empty submissions.

BYPASSABLE
Server-Side (Flask)

Security Mandatory

Cannot be bypassed. Protects your database and business logic.

NON-NEGOTIABLE

CRITICAL: CSRF Protection

Cross-Site Request Forgery (CSRF) is an attack where a user is tricked into performing actions on your site without their consent while they are logged in.

How it attacks?

It exploits the browser's behavior of automatically sending cookies with requests.

How we stop it?

Flask-WTF injects a unique, random CSRF Token that must match on every POST request.

4 Common Beginner Mistakes

  • Trusting frontend validation as your only line of defense.
  • Saving data to the database before running form.validate_on_submit().
  • Failing silently instead of showing specific errors to the user.

Day 8 Checkpoint

  • Understand why request.form is untrusted
  • Explain the role of Flask-WTF as a safety layer
  • Define CSRF and how tokens prevent it
  • Realize that validation is part of UX and Security
