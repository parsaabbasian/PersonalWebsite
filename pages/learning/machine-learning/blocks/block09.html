Day 18

REST API Design

Learning the "Architecture of the Internet." Design professional APIs that are stateless, scalable, and easy to consume.

1 Resources, Not Actions

In REST, the URL should always identify a Noun (the thing), never a Verb (the action). You use the HTTP Method to describe the action.

❌ Action-Based (Bad)

/getUser/5
/deletePost/12
/createComment

✅ Resource-Based (REST)

/users/5
/posts/12
/comments

2 The Vocabulary of HTTP

REST leverages the existing architecture of the web. By using standard methods, any client (mobile app, browser, or server) knows exactly what to expect.

Method Database (CRUD) Intent
GET Read Fetch a resource
POST Create Add a new resource
PATCH Update Modify a resource
DELETE Delete Remove a resource

3 Statelessness: The Key to Scale

A REST API is Stateless. The server does not store "sessions." Every request from the client must contain all the information necessary to understand and authorize that request (usually via a Token).

"Statelessness allows us to spin up 100 servers, and a request can go to any one of them and still work."

4 The API's Language

In REST, we use status codes to communicate result states. An API should never return a "200 OK" if the actual result was an error.

201 Created

Returned after a successful POST request.

403 Forbidden

User is authenticated but lacks permission.
