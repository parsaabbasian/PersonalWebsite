Day 11

Database Migrations

Version control for your data structure. Learn to evolve your database without losing user data.

1 Why Schema Changes Are Dangerous

A Schema is the blueprint of your database (tables, columns, types). When you change your code (e.g., adding a "bio" column to a User), the database doesn't magically update.

The Production Risk

In production, you can't just delete the database and start over. Real users have real data. If you change a data type incorrectly, you could corrupt or lose every record in that table.

2 Migrations = Version Control

Think of migrations as Git for your database structure. Each migration is a "commit" that tracks a specific change.

VERSION 1

ID, Username, Password

VERSION 2

+ Added Email column

VERSION 3

+ Renamed Password to Hash

3 Recreating vs. Migrating

Recreating (❌)

Deletes all existing data. Breaks foreign key references. Fine for local demos, fatal for real apps.

Migrating (✅)

Preserves data. Applies changes incrementally. Can be reviewed and shared with teammates.

4 The Migration Workflow

Upgrade

Moves the schema forward. (e.g., Adding a 'created_at' timestamp).

Downgrade

Reverses a change. (Warning: some changes, like deleting a column, are harder to reverse safely).

5 Common Beginner Mistakes

  • Assuming db.create_all() handles future changes. (It only creates missing tables).
  • Deleting migration files manually to "clean up" the project.
  • Applying migrations in production without a database backup.

Day 11 Checkpoint

You have shifted from a "local coder" to a "backend engineer." You should now understand:

  • What a Database Schema is
  • Why migrations protect user data
  • The role of Flask-Migrate
  • The Upgrade/Downgrade cycle
