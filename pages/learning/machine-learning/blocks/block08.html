Day 17

Authorization & Roles

Mastering access control: the "What can you do?" of backend engineering.

1 Authentication vs. Authorization

It is a common mistake to think that because a user is "Logged In," they have permission to do everything. In professional backends, we treat these as two entirely separate gates.

Authentication

"Who are you?" (Passport/Login)

Authorization

"What can you do?" (Clearance/Permissions)

2 The 3 Core Questions

Every time a user tries to access a page or modify data, the backend must pass three sequential tests:

Step 1

Is the user logged in? (Authentication)

Step 2

Does the user have the right role? (Admin vs. User)

Step 3

Does the user own this data? (Ownership)

3 Roles vs. Permissions

Model Context Example
Roles High-level categories admin, editor, viewer
Permissions Specific granular actions can_delete_user, can_edit_post

Critical Threat: Privilege Escalation

This happens when a normal user finds a way to perform admin actions. Usually because the developer checked the "Login" status but forgot to check the "Role" or "Ownership."

Backend Secret: "Deny by Default." If you don't explicitly grant a user permission to do something, the server should say no. Never allow access unless proven otherwise.
