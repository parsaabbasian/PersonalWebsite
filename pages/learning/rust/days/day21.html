Day 21

Mini Project 2: Secure REST API

Architecting a production-style backend. This is where JWT, REST, and Ownership Logic meet.

1 Defining the Resource

In API development, we don't think in pages; we think in Resources. For this project, a "Note" is a resource that belongs to a specific user.

# The Resource Schema (JSON)
{
"id": 101,
"title": "Secure Backend",
"owner_id": 5,
"created_at": "2026-01-07"
}

2 The Endpoint Map

Method Endpoint Auth
POST /auth/login Public
GET /notes JWT Required
PATCH /notes/<id> Owner Only
DELETE /notes/<id> Owner Only

3 Enforcing Ownership

It’s not enough to be logged in. Your API must verify that User A cannot edit User B’s data. This logic happens at the view level.

Security Checklist

  • • Is the token valid and not expired?
  • • Does the resource ID exist in the database?
  • • Does the resource's owner_id match the JWT sub?

Project Deliverable

By the end of this day, your backend should be a pure Data Service. No HTML, no flashes—just precise JSON and status codes.

Success Response
200 OK + { "data": [...] }
Auth Error
401 Unauthorized + { "err": "..." }
